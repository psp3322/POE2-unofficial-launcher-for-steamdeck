# 폰트 변조 분석 보고서 — 원본 vs 게임 적용 샘플 (11종 전수)

> 작성일: 2026-05-17
> 분석 대상: 로컬 원본(`gh-pages/fonts`) ↔ 배포자 GDrive 변조본(`test/samples`) 11종
> 도구: 프로젝트 내장 `fonteditor-core`로 직접 디코드 (추정 아님, 실측)
> 변조본 명명 규칙(실측 확정): **`_1` = Noto Sans CJK TC**, **`_2` = Spoqa Han Sans Neo**

---

## 0. 요약 (TL;DR)

1. **글리프(글자 모양)는 원본과 동일.** 변조본은 글자 모양을 안 건드린다. 글자 크기 문제는 글리프가 아니라 **세로 metrics(ascent/descent)** 때문이다.
2. **현재 프로젝트 코드는 name table 6필드만 변조.** 변조본은 그 외에 **name table 전체 + head/OS2 스타일 비트 + hhea/OS2 metrics**까지 바꾼다.
3. **글자 크기 핵심 = `hhea.ascent` / `OS/2.usWinAscent`.** 원본마다 작아서(예: Kostar 800), 현재 코드처럼 metrics를 안 바꾸면 게임에 작은 폰트가 등록돼 크기가 어긋난다.
4. **고정 복제 필드 확정**: Noto/Spoqa 변조본은 원본과 무관하게 일부 metrics를 고정값으로 덮어쓴다 (`tAsc`, `tDesc`, `fsSel`, `wt` 등 — 5장 참조).
5. **사용자 가설("시간에 따라 수치 조정") 데이터로 입증.** 같은 타겟(Noto)인데 `hAsc`/`wAsc`가 **4개 그룹(986 / 1035 / 1044 / 873)으로 뭉친다.** 단순 복제라면 단일값이어야 한다 → 배포자가 작업 시기별로 ascent 기준을 바꿔 재배포한 흔적.
6. **round-trip 검증 완료**: `fonteditor-core`로 변조 없이 read→write 시 metrics 불변 → 변조본의 차이는 라이브러리 부수효과가 아니라 **배포자가 의도적으로 넣은 값**.

## 2. 글리프는 동일 — 크기 문제의 원인이 아님

한글 '가'(U+AC00) 글리프 bounding box를 원본↔변조본 비교 시 **xMin/yMin/xMax/yMax/advanceWidth 전부 동일**(Kostar·DNF 검증). 변조본은 글자 모양을 손대지 않는다. → 크기 문제는 글리프 스케일이 아니라 **폰트가 게임에 보고하는 세로 공간(metrics)** 이다 (4장 metrics 상세 참조).

## 3. Name Table 변조 — 폰트 신원 정보 일괄 교체

> Name table은 폰트의 "이름표". 게임/Windows는 이 ID들을 읽어 폰트를 식별한다. 변조본은 **타겟 폰트(Noto/Spoqa)의 name table을 거의 통째로 복제**한다 — 즉 신원을 완전히 갈아끼운다.

### 3.1 각 Name ID의 역할과 게임 인식 영향도

| Name ID | 필드(fonteditor-core) | 역할               | 게임 인식 영향                       |
| ------- | --------------------- | ------------------ | ------------------------------------ |
| 0       | copyright             | 저작권 문구        | 없음 (표시용)                        |
| 1       | fontFamily            | **패밀리 이름**    | **★ 높음** — 폰트 검색 1차 키        |
| 2       | fontSubFamily         | 스타일(Regular…)   | 중간                                 |
| 3       | uniqueSubFamily       | 고유 식별자        | 낮음~중간 (중복 등록 충돌 회피)      |
| 4       | fullName              | **전체 표시 이름** | **★ 높음** — 일부 엔진이 이걸로 매칭 |

### 3.3 관찰

1. **변조본은 name table 전체를 타겟 폰트로 통째 복제**한다. copyright·designer·licence·URL까지 전부 Noto/Spoqa 원문 — 게임 인식과 무관한 표시용 필드(ID 0, 7~14)까지 베낀 "정품처럼 보이게" 군더더기.
2. **게임 인식에 실제 필요한 건 ID 1/4/6 (+16)** 정도. 현재 런처 코드는 이 핵심 필드는 변조하므로 **게임 인식 자체는 동작**한다.
3. **현재 런처 코드의 결함 2가지**:
   - ID 3(uniqueSubFamily)을 `postScript;원본version`으로 **잘못 합성** — 타겟 원문과 다름. 충돌 가능성 낮으나 부정확.
   - ID 16/17(preferredFamily) 주입을 시도하지만, 실제 샘플은 16/17이 **비어 있음**(`(없음)`). 즉 타겟 폰트엔 16/17이 없는데 런처는 채워 넣어 **타겟과 불일치** 발생.
4. **Noto의 fontSubFamily=`Book`** 인데 fontFamily엔 Book 없음 — "윈도우 제목엔 Book 빼고, 엔진 인식엔 Book 넣는" 의도. 현재 코드도 이 부분은 맞게 처리 중.
5. **결론**: name table 변조는 글자 크기와 무관(크기는 4·5장 metrics). 단, ID 3/16/17 처리가 타겟과 어긋나므로 자동화 시 정정 필요.

### 5.2 시기별로 갈리는 필드 (`hAsc`/`wAsc`) — 사용자 가설 입증

같은 타겟인데 ascent가 단일값이 아니라 **명확한 그룹**으로 뭉친다:

**Noto(\_1) `hAsc` 그룹**

| hAsc | wAsc | 해당 폰트                             | 추정 작업 시기 |
| ---- | ---- | ------------------------------------- | -------------- |
| 986  | 986  | Kostar, 경기천년바탕                  | 그룹 A         |
| 1035 | 1035 | DNF-M, 빙그레체, 프리텐다드           | 그룹 B         |
| 1044 | 1044 | DNF-L, 이사만루, 메이플, 워헤이븐 B/L | 그룹 C         |
| 873  | 990  | Kodia (단독, hAsc≠wAsc)               | 그룹 D (특수)  |

**Spoqa(\_2) `hAsc` 그룹**

| hAsc | wAsc | 해당 폰트                             |
| ---- | ---- | ------------------------------------- |
| 1102 | 1102 | Kostar, 경기천년바탕                  |
| 1208 | 1208 | DNF-M, 빙그레체, 프리텐다드           |
| 1160 | 1160 | DNF-L, 이사만루, 메이플, 워헤이븐 B/L |
| 1018 | 1155 | Kodia (단독)                          |

**해석**:

- 단순 복제였다면 모든 Noto 변조본의 hAsc가 단일값이어야 한다. **4개 그룹으로 나뉘는 건 배포자가 작업 시기별로 ascent 기준을 바꿔가며 재배포했다는 사용자 가설을 데이터로 입증**한다.
- 그룹 A(986) → B(1035) → C(1044) 로 **점진적으로 값을 키운** 흐름이 보인다. (Kostar/경기천년바탕은 초기 작업, 워헤이븐/메이플 등은 후기 작업으로 추정)
- **Kodia(그룹 D)만 `hAsc≠wAsc`** — 가장 정교한 최신 조정 또는 별도 도구. 게임이 두 필드 중 무엇을 읽는지 확정하는 데 결정적.

## 8. ⭐ 예외 케이스: 런처 생성 폰트가 과하게 커 보이는 이유

> 사용자 관찰: "내 런처로 만든 폰트를 설치하면 글자가 과하게 크고, 배포자 샘플은 정상 크기다."

### 8.1 3자 비교 (Kostar→Noto 기준, 실측)

| 필드                    | 원본 Kostar | **런처 생성**    | 샘플(정상) |
| ----------------------- | ----------- | ---------------- | ---------- |
| hhea.ascent             | 800         | **800 (그대로)** | 986        |
| OS/2.usWinAscent        | 800         | **800 (그대로)** | 986        |
| OS/2.sTypoAscender      | 800         | **800 (그대로)** | 880        |
| 글리프 '가' 실제높이    | 836         | 836 (동일)       | 836 (동일) |
| 신고 라인높이(asc−desc) | 1000        | **1000**         | 1230       |
| 글리프/라인 비율        | 0.836       | **0.836**        | 0.680      |

→ 현재 런처([FontMutatorWorker.ts:37-45](src/main/workers/FontMutatorWorker.ts#L37-L45))는 name table 6필드만 바꾸고 **metrics를 전혀 손대지 않는다.** 원본 Kostar의 `ascent=800`이 그대로 남는다.

### 8.2 왜 커 보이나 — 메커니즘

게임은 한 줄 칸에 **폰트가 신고한 라인높이(ascent−descent)**를 꽉 채우고, 그 안에서 글리프가 차지하는 **비율**대로 글자를 그린다.

- **글리프 크기는 셋 다 동일(836).** "폰트가 커서"가 아니다.
- 런처 폰트: 라인높이를 **1000**으로 신고 → 글리프가 칸의 **83.6%** 차지 → **크게 보임**
- 샘플: 라인높이를 **1230**으로 신고(ascent를 부풀림) → 글리프가 칸의 **68%** 차지 → 정상 크기

즉 정확한 원인은 "폰트가 크다"가 아니라 **"ascent를 안 키워서 글리프가 라인 칸에서 과도한 비율을 차지한다"** 이다. 배포자는 ascent를 의도적으로 뻥튀기해 비율을 낮춰 정상 크기로 맞춘 것.

### 8.3 Kodia 예외 — 다른 변조 파이프라인 (원본 결함 아님)

Kodia는 다른 10종과 **변조 방식 자체가 다르다.** 사용자 제공 원본은 정상이며, 차이는 전적으로 배포자의 변조 단계에서 발생한다.

| 항목            | 일반 패턴(Kostar 등 10종)            | Kodia                                                       |
| --------------- | ------------------------------------ | ----------------------------------------------------------- |
| 글리프          | 원본 그대로 보존 (좌표·폭·개수 동일) | **글자를 ~1.17배 확대해 다시 그림** (adv 922→1078, glyf +1) |
| tAsc/tDesc/tGap | 880 / -120 / 0 고정                  | **970 / -282 / 80** (Kodia만)                               |
| mac             | 1 (Bold비트, 실수성)                 | **0** (원본 유지, 정상)                                     |
| hAsc vs wAsc    | 동일 (986=986)                       | **다름** (873 ≠ 990)                                        |

`kodia.ttf` 원본 `version`은 `February 23, 2026;FontCreator 15.0.0` — FontCreator로 만든 커스텀 폰트(디아블로2 한글)이며 **손상된 파일이 아니다.** 배포자가 이 폰트는 일괄 스크립트(이름표·metrics만 덮어쓰기)로 크기가 안 맞아서, **FontCreator GUI로 글리프까지 직접 확대 보정**한 것으로 추정. → "예외"가 아니라 **수동 보정 파이프라인** 케이스. 따라서 7장의 "Kodia로 hhea/wAsc 검증" 항목은 글리프가 변형되어 순수 대조군이 될 수 없음(정정).

### 8.4 자동화 설계 — ascent 계산 주입 (채택 방향)

배포자의 ascent 값은 일관 공식이 아니다. 글리프높이/라인높이 비율이 0.667~0.843로 들쭉날쭉 → 눈대중 작업 흔적. **이를 흉내내지 말고 코드가 비율을 계산**해야 폰트 종류 무관하게 일관된다.

추가 실측: 샘플의 `tAsc−tDesc`(typo 라인높이)는 그룹별 **고정값**(그룹 A/B = 1000, 그룹 C/Kodia = 1252). 즉 배포자는 typo 라인높이는 타겟별 상수로 두고, hhea/win ascent만 폰트별로 흔들었다.

**자동화 원리**: 게임에서 "정상 크기" = 타겟 폰트(Noto/Spoqa)와 동일한 **글리프높이 ÷ 라인높이 비율**. 워커가:

1. 원본 폰트의 대표 글리프 실측 높이(또는 글리프 bbox 통계)를 측정
2. 목표 비율(정품 Noto/Spoqa의 글리프높이/라인높이, 또는 검증으로 확정한 상수)에 맞춰 `hhea.ascent`/`usWinAscent`를 **역산**하여 주입
3. typo 라인높이·fsSel·mac·wt 등은 5장에서 확정한 고정 상수로 주입
4. 글리프는 건드리지 않음 (Kodia식 확대는 채택 안 함 — metrics 조정으로 동일 효과 달성)

### 8.5 폰트별 비율 실측 (보정 임계값 근거)

`r = 한글 글자높이('한' U+D55C) ÷ 라인높이`. upm 1000 정규화. **r이 클수록 게임에서 크게 보임 → 보정 필요.**

| 폰트       | 원본 r (=런처) | 샘플 Noto r | 샘플 Spoqa r | 원본→샘플 거동         |
| ---------- | -------------- | ----------- | ------------ | ---------------------- |
| 경기천년   | **0.935**      | 0.760       | 0.680        | 강하게 줄임 (필수)     |
| Kostar     | **0.857**      | 0.697       | 0.623        | 강하게 줄임            |
| 메이플     | 0.833          | 0.806       | 0.691        | 약간 줄임              |
| 프리텐다드 | **0.820**      | 0.711       | 0.711        | 줄임                   |
| Kodia      | 0.806          | 0.775       | 0.664        | 약간 줄임(글리프 확대) |
| 이사만루   | 0.759          | 0.766       | 0.661        | 거의 유지              |
| DNF-M      | 0.743          | 0.782       | 0.675        | 거의 유지              |
| 빙그레     | 0.739          | 0.807       | 0.686        | 오히려 키움            |
| DNF-L      | 0.731          | 0.770       | 0.665        | 거의 유지              |
| 워헤이븐 B | **0.691**      | 0.684       | 0.615        | **거의 무보정**        |
| 워헤이븐 L | **0.690**      | 0.683       | 0.615        | **거의 무보정**        |

(샘플 Noto/Spoqa r은 `usWinAscent` 기준값. 이사만루·메이플·Kodia는 글리프도 확대돼 hhea 기준값은 별도 — 8.3 참조)

**관찰**:

- 샘플(정상)이 수렴하는 목표 비율: **Noto rWin ≈ 0.68~0.81 (평균 ~0.74)**, **Spoqa rWin ≈ 0.61~0.71 (평균 ~0.66)**. 값(ascent)은 들쭉날쭉해도 결과 비율은 좁은 범위로 수렴 → 자동화 목표값 후보.
- **원본 r이 낮은 폰트(워헤이븐 ≈0.69)는 원본과 샘플 비율이 거의 동일** (0.690 → 0.684). 배포자도 사실상 보정 안 함. → **"특정 비율 이상일 때만 보정" 임계값 방식이 데이터로 검증됨.**
- 원본 r이 높은 폰트(경기천년 0.935, Kostar 0.857)는 샘플에서 강하게 축소됨 → 보정 효과가 큰 그룹.

### 8.6 자동화 설계 (threshold + 목표비율)

```
원본 r = 한글글자높이 / (usWinAscent - usWinDescent)   // upm 1000 정규화

if (원본 r > THRESHOLD) {
    목표 라인높이 = 한글글자높이 / TARGET_RATIO
    → usWinAscent / hhea.ascent 를 역산 주입 (descent 비율 유지)
} else {
    metrics 보정 생략 (원본 유지)   // 워헤이븐 같은 이미-적정 폰트
}
// fsSel/mac/wt/typo 라인높이/name table 은 r과 무관하게 항상 고정 상수 주입
```

### 9.2 결론 — 분석/설계 전환

1. **샘플은 정답이 아니었다.** 배포자 name table 복제는 정확하나 **metrics는 본체와 불일치**. 특히 Noto는 샘플이 본체보다 글자가 큼(r 0.697 vs 0.611). 4·5장의 "샘플 그룹/평균"과 8장 "45쌍 평균 상수(0.742/0.651)"는 **참고용으로 강등** — 진짜 기준은 본체 metrics.
2. **본체 Spoqa는 `hhea ≠ win`** (hhea 970, win 1100). 본체 Noto는 hhea=win=1160. → 자동화 시 두 필드를 본체값 그대로 분리 주입해야 함.
3. **자동화 대폭 단순화**: 비율 역산·threshold·평균계산 **전부 불필요**. 게임 본체 metrics를 **고정 상수로 그대로 주입**하면 글리프가 게임 폰트와 동일 비율로 렌더됨. (배포자가 눈대중으로 하던 걸 정답값으로 대체)

### 9.4 크기 슬라이더 설계 공식 (검증 완료)

본체 metrics = **슬라이더 100%**. scale%로 글자 크기를 비례 조정:

```
ascRatio = 본체ascent / (본체ascent - 본체descent)      // descent는 음수
baseLine = 본체ascent - 본체descent

newLine  = round(baseLine * 100 / scale)                // 글자 ∝ 1/라인높이
ascent   = round(newLine * ascRatio)
descent  = ascent - newLine                              // 음수
// usWinAscent/Descent 도 동일 비례. typo/fsSel/wt/mac/name 은 scale 무관 본체 상수 고정.
```

검증 결과(본체 Noto, 글리프 884 기준): 슬라이더 50→화면 50%, 100→100%, 150→화면 150%로 **정확히 선형 일치.** (8장 부록 재현 가능)

→ 범위 50~150%, 기본 100%. Noto/Spoqa 독립 슬라이더.

## 10. ⭐ 배포자 변조 작업의 전모 (메이플 폰트 실측 + 커뮤니티 출처)

> 출처: dcinside POE 갤러리 글(no=960339) + 사용자 조사 문서(`poe-font-role-research.md`, 동 폴더). 게시글이 주장한 3대 작업을 메이플 L 원본↔변조본으로 실측 검증.

배포자 본인이 밝힌 작업: **① 크기 조정 ② 위치 보정(윗 공백 → 글자를 아래로) ③ 외계어(글리프) 추가**. "폰트마다 크기·위치값이 달라 개별 조정 + 인게임 반복 테스트"라고 명시 → 4·5장에서 관찰한 ascent 그룹 산포(눈대중)의 직접적 근거.

### 10.1 메이플 L 실측 (원본 → 샘플 \_1 Noto)

| 항목                   | 원본 Maplestory L | 샘플 \_1 (Noto) | 변조 작업             |
| ---------------------- | ----------------- | --------------- | --------------------- |
| hhea asc/desc          | 850 / -265        | 873 / -253      | metrics 조정          |
| win asc/desc           | 850 / 265         | 990 / 279       | metrics 조정          |
| numGlyphs              | 12261             | 12259           | 글리프 재구성         |

### 10.2 게시글 3대 작업 — 전부 실측 확인

1. **크기 조정 = 글리프 ~10% 확대.** 한글·영문 모든 글자가 균일하게 ×1.10 (Kostar류 metrics-only 변조와 다른, Kodia식 글리프 변형 파이프라인). 메이플 폴더명 `메이플 L (외계어 지원 추가)`이 이를 뒷받침.
2. **위치 보정 = 글리프 중심 y +62 이동.** 모든 글자 중심이 균일하게 위로(+y) 평행이동 → 게시글의 "윗부분 공백 생김 → 글자를 (시각적으로) 중앙/아래로" 보정과 정확히 일치. 단순 metrics가 아니라 **글리프 좌표 자체를 평행이동**.
3. **외계어 추가 = 글리프 재구성.** numGlyphs/cmap이 원본과 달라짐(12261→12259, cmap +5). 단, 테스트한 특수기호(➀ U+2780, ☠ U+2620 등) 커버는 원본·변조본 동일 → "외계어"는 게임이 쓰는 특정 누락 글리프 보충으로 추정(범위 추가 조사 필요).

### 10.3 자동화 설계에 주는 함의 (중요)

- 9.4의 슬라이더 공식은 **metrics 조정만** 다룬다. 그러나 배포자는 일부 폰트(메이플·Kodia)에서 **글리프 확대 + y이동 + 글리프 추가**까지 했다 — metrics만으론 재현 불가능한 영역.
- **단, 본체 폰트(9장 정답)가 있으므로 글리프 확대/이동은 불필요.** 글리프를 건드리는 건 배포자가 "정답 metrics"를 몰라서 눈대중으로 보정하다 한계에 부딪혀 글리프까지 손댄 것. 우리는 본체 metrics를 정확히 주입하므로 **글리프 무변형 + metrics 상수 주입**으로 동일한 결과를 얻는다 (계획서 방침 유지 타당함을 재확인).
- **"외계어 추가"만 별도 검토 대상**: 게임이 요구하는데 사용자 커스텀 폰트에 없는 글리프가 있다면, metrics 주입만으론 해당 글자가 깨질(두부) 수 있음. → 계획서 8장 미확정에 "누락 글리프 폴백" 항목 추가 필요.

## 11. 부록 — 검증 재현 방법

> 분석에 쓴 일회성 스크립트(`compare_all.mjs`, `stats.mjs`, `roundtrip.mjs`,
> `analyze_samples.mjs`, `verify_mutation.mjs`)는 **커밋하지 않았다**(scratch
> 작업물, 상용 폰트/게임 추출물 의존으로 리포 보관 부적합). 아래 방법론으로
> 프로젝트 내장 `fonteditor-core`를 써서 재작성 가능하다.

- **폰트별 metrics 재추출 / 그룹 통계**: `Font.create(buf,{type:"ttf"}).get()`
  으로 `head.unitsPerEm`, `hhea.ascent/descent`, `OS/2.usWin*·sTypo*·fsSelection`
  추출. 한글 '한'(U+D55C) 글리프 bbox로 `r = 글자높이/라인높이` 산출 → 본
  보고서 4·5·7장 표 재현.
- **round-trip 부수효과 검증**: 원본을 변조 없이 `font.write()`→재로드,
  head/OS2/hhea 불변 확인 (9장에서 변조본 차이가 라이브러리 부수효과가 아님을
  입증한 방법).
- **원본↔샘플 글리프 매핑**: 주요 글자 bbox/advanceWidth를 md5 해시해 로컬
  원본과 대조 (이름 달라도 글리프 같으면 동일 원본).
- **변조 산출물 검증**: `FontMutatorWorker.ts` 로직을 재현해 변조 후
  head/hhea/OS2/name을 9.3 본체 상수와 필드 단위 assert (STEP 2 검증에서
  win descent 부호 버그를 잡은 방법).
